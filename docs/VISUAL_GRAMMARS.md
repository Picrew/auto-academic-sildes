# Visual Grammars

Use visual grammars to avoid one repeated template. A grammar is not a theme
skin; it controls slide rhythm, evidence scale, typography, and what kind of
proof gets visual weight.

## Editorial Profile

Use for public researchers, labs, portfolios, and biographies.

- calm editorial hierarchy
- small identity surface on cover
- large homepage/project proof surfaces later
- restrained serif headline
- synthesis slide at the end

Template: `templates/editorial-profile/DESIGN.md`

Use for systems, infrastructure, benchmarks, tools, APIs, and engineering talks.

- modular grid

- strong alignment

- clear labels and routes

Template: `templates/swiss-systems/DESIGN.md`

## Academic Homepage Grid

Use for researcher, lab, or project profiles that should feel like a polished
academic homepage: clean, dense, source-linked, and useful at a glance.

- left identity/source rail plus right content field
- publication/project cards instead of generic feature blocks
- serif headings with Inter-like sans body

## Prism Dossier

## Raw Grid Research

- strong poster covers, stat boards, and comparison ledgers
- content should feel direct, source-backed, and deliberately un-smooth
- proof images sit in hard frames; tiny screenshots are not accepted

## Renderer Checks

Every grammar should make these visible in the output:

- each slide should expose a named `data-composition`, such as `cover-source-rail`, `proof-stage`, `proof-atlas-spread`, `artifact-dossier`, or `matrix-ledger`
- proof pages give the evidence image enough visible source-pixel area to inspect; `proof-image-small` warnings should lower the grammar ranking even when they do not block export
- use explicit safety intents such as `proof-showcase` or `artifact-showcase` when automatic grammar selection produces small screenshots, crowded side notes, or letterboxed proof windows; these intents resolve to grammar-specific compositions rather than one fixed template
- metric cells must render as two clear lines: value first, label second; cramped value/label collisions are hard failures, even when they look like minor typography
- callout pins must be attached to the actual rendered source-image box, not to the surrounding proof slot; pins in letterbox whitespace are a failed annotation
- proof and artifact images are judged by visible source-pixel area, not by the size of the containing frame
- tiny cover images are flagged as decorative identity anchors; they should not be counted as proof
- overlong headings, subtitle paragraphs, bullets, labels, captions, and metric labels should be rejected by `check` before visual rendering
- one layout composition should not repeat more than two slides in a row
- neutral base plus one or two accents should replace one-note palettes
- section cadence should include scale contrast: large proof, compact map, then synthesis
- callouts should be one to three pins with short edge notes
- ordinary content slides should fill the canvas with meaningful grouped content, not tiny text surrounded by unused space

## Anti-Sameness Knobs

Vary these across decks:

- cover composition: full title, split source surface, or quiet title card
- evidence layout: left notes, right notes, full bleed, or atlas spread
- safety override: `proof-showcase` for a dominant figure/screenshot, `artifact-showcase` for a readable source panel; both are semantic requests for proof scale, not a demand that every grammar share the same layout
- cadence: dark/light alternation or section bands
- typography scale: editorial, compact, or poster-like
- proof object type: figure, trace, homepage, leaderboard, repo, UI state
- grammar family: academic homepage, scientific memo, typographic manifesto, or evidence wall
- composition profile: select a different relationship between claim, evidence, and notes, not just a different accent color

## Validated Selection Notes

The historic 40-variant strict bake-off across the Fixture A and Fixture B
profile fixtures suggests this practical ranking. The default pool now contains
42 grammars, and the latest focused hardening passes added
`signal-intelligence-brief`, `raw-grid-research`, `stencil-field-tablet`,
`maison-research-catalog`, `folio-swiss-noir`, and
`chromatic-research-map` alongside earlier validated
`prism-workbench-index`, `huashu-issue-broadsheet`, `prism-newsroom-index`, and
`js-editorial-cascade` on both fixtures with zero blocking issues.

- High-taste default pool: `forest-editorial-brief`, `object-study-wall`,
  `evidence-atelier`, `cobalt-research-grid`, `prism-clean-room`, and
  `takram-research-system`. These should be judged by proof scale, visual
  rhythm, and non-generic taste.
- Conservative profile defaults: `academic-homepage-grid`, `prism-dossier`, and
  `prism-clean-room`. They are clean, source-linked, and least likely to
  overpower public-profile evidence.
- Scholarly special case: `atlas-marginalia`. It has the best academic-page
  taste when the source material has enough homepage, paper, or project
  evidence; sparse profiles can look underfilled.
- Systems special cases: `systems-field-manual`, `lab-trace-ledger`, and
  `cobalt-research-grid`. Use them for tool-use, infrastructure, benchmarks,
  APIs, engineering systems, and agent/RL traces.
- Formal/handout special cases: `vellum-research-note`, `mono-ink-ledger`,
  `ia-research-archive`, `monograph-review`, `paper-atlas`, `catalog-atelier`, and
  `fathom-research-brief`. Use them when the output should feel like a dossier,
  paper note, or source-linked review; watch `proof-image-small` warnings.
- Cartographic/project special case: `stamen-data-map`. Use it when source
  artifacts form a project, benchmark, or ecosystem landscape.
- Web/editorial special cases: `js-editorial-cascade` for asymmetric HTML-like
  proof staging, and `sumi-research-scroll` for quiet source-indexed academic
  reading.
- Template-inspired special cases: `signal-intelligence-brief` for
  source-heavy institutional dossiers, `raw-grid-research` for hard-edged
  benchmark/demo boards, and `stencil-field-tablet` for archival field-manual
  slides where the artifact should feel inspected rather than decorated.

Use extra caution with `jetset-theory-grid`, `keynote-evidence-wall`, and
`broadside-lab`, `pentagram-gridnote`, and `neo-grid-lab`: they are useful as
deliberate typographic or demo resets, but can quickly become theatrical or
drown out evidence. Use `swiss-systems` as a quiet fallback only when its
structure adds information beyond a palette change.

Evidence discipline matters more than grammar choice:

- Empty label, metric, or bento modules are forbidden; remove the box or give it
  a real project, metric, source, or citation.
- The same artifact should not appear more than twice unless the crop and slide
  job change materially.
- Portraits and logos are identity anchors, not research evidence.
- Homepage screenshots are useful for identity and positioning; research claims
  still need project pages, repos, paper figures, result tables, or benchmark
  surfaces.

Do not vary by adding decoration. Vary by changing the relationship between
claim, evidence, and whitespace.

## Grammar Bake-Off

Use `compare-grammars` when a brief is vague or when a deck starts to look like
one repeated template:

```bash
uv run academic-deck compare-grammars \
  --deck examples/starter/deck.yaml \
  --out outputs/fixture-a
```

For the high-sense anti-template workflow, use the curated 20-pass reference
preset before trying the full default sweep:

```bash
uv run academic-deck compare-grammars \
  --deck examples/starter/deck.yaml \
  --out outputs/fixture-a \
  --grammars highsense-20 \
  --fail-on-layout
```

`highsense-20` includes academic-homepage, source-archive, Huashu hard-board,
Beautiful HTML Templates families (`signal`, `raw-grid`, `stencil-tablet`),
gallery proof, web-editorial, and newer Maison/Folio/Chromatic families so the
first review compares composition schools rather than near-identical palettes.

When the brief is tone-led rather than grammar-led, generate a template scan
first:

```bash
uv run academic-deck template-shortlist \
  --brief "academic research profile, source evidence, quiet high taste" \
  --out outputs/template-shortlist
```

The command renders multiple grammar variants, writes a stacked overview image,
records each composition sequence in `GRAMMAR_COMPARISON.md`, and writes
`DESIGN_DIRECTIONS.md` as a higher-level design-direction brief. The direction
report maps clean variants into 20+ schools such as PRISM selected-work indexes,
Huashu issue broadsheets, Fathom evidence briefs, gallery proof rooms, and
source archives. Use it to choose a layout philosophy before choosing a final
grammar. In comparison mode, `--fail-on-layout` fails only when no ready
zero-error / zero-strict-warning variant remains; rejected variants stay in the
report as repair evidence. The comparison report scores each grammar with
conservative penalties for proof images that are too small, artifact panels
that cannot be read, letterboxed source images, and cover thumbnails that are
decorative only.
Prefer the `Recommended Shortlist`; rows with decorative cover warnings should
have the cover crop adjusted before final export. Use the `Image Revision Queue`
as a to-fix list, not as an accept list. Read `Repair Hints` or generate
`DECK_REPAIR_PLAN.md` from `GRAMMAR_REPAIR_HINTS.json` for slide-specific crop,
layout, or split-slide actions before rerendering. Do not choose a variant
merely because the palette is different.

Every visible box needs a job: evidence, comparison axis, timeline/genealogy,
method component, or conclusion anchor. Empty modules and generic cards are AI
flavor even when the layout technically passes.

Renderer budgets are now part of the grammar contract. A slide should not
